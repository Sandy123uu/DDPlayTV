## Context

当前 SonarCloud 报告（`.sonarcloud-report`）显示质量门为 `ERROR`，阻塞项集中在：

- `new_coverage = 0.0`（阈值 80）
- `new_duplicated_lines_density = 3.6`（阈值 3）
- `new_security_hotspots_reviewed = 0.0`（阈值 100）

同时存在 3 个漏洞、52 个 Security Hotspot、978 个未解决 Issue。问题分布高度集中在少量文件（例如 `ControlWrapper.kt`、`NumberPicker.java`、多种字幕格式解析类），并以以下规则类型为主：

- 复杂度超限（`S3776`）
- `switch` 缺少 `default`（`S131`）
- 安全实现问题（如不安全加密模式、release 构建混淆配置）

本变更是“质量治理变更”，需要在不破坏业务行为的前提下，完成一次可回归、可追踪的集中修复，并让质量门恢复到可通过状态。

## Goals / Non-Goals

**Goals:**

- 以质量门阻塞项为第一优先级，恢复 SonarCloud 质量门可通过。
- 优先修复安全问题（漏洞 + Hotspot 处理），并补齐必要审查记录。
- 对高频高严重规则进行结构化重构，优先处理问题密集文件。
- 为新增/改动逻辑补充最小必要测试，提升 new code 覆盖率并防止回归。
- 建立可追踪的治理清单（问题来源、修复提交、验证结果）。

**Non-Goals:**

- 不追求在单次变更内清零全部 978 个历史问题。
- 不进行与质量修复无关的大规模功能重写或架构迁移。
- 不引入新的跨 feature 直接依赖，保持现有模块分层规则，遵守已有模块的功能方法边界，遵守软件工程设计规范。

## Decisions

### 1) 修复顺序采用“质量门阻塞项优先”的分层策略

- 先处理直接导致质量门失败的三个指标（覆盖率、重复率、Hotspot 审查率）。
- 与指标相关的代码修复按“安全 > 高严重复杂度 > 高频规范问题”排序。
- 采用小批次提交（每批可独立验证），避免一次性大改难以回滚。

**备选方案**：按模块一次性清理全部问题。  
**不选原因**：变更面过大、风险不可控，且难以快速恢复质量门。

### 2) 安全问题以“代码修复 + 配置硬化 + 审查闭环”并行推进

- 针对 `EntropyUtils.kt` 的不安全加密实现，升级为安全模式（优先 AEAD 方案）并保持调用方兼容。
- 针对 `app/build.gradle.kts` 的 release 混淆告警，明确开启并验证 release 混淆链路。
- Security Hotspot 不仅修代码，还同步完成审查与结论沉淀，确保 `new_security_hotspots_reviewed` 达标。

**备选方案**：仅在 SonarCloud 标记热点为“已审查”但不改代码。  
**不选原因**：无法降低真实安全风险，且后续审计不可持续。

### 3) 高复杂度方法采用“提炼子函数 + 守卫式返回 + 语义分层”重构

- 对 `S3776` 集中的方法（如 `AnimeSeasonActivity.initRv`、`NumberPicker` 的事件分发方法）按行为分支拆分为小函数。
- 在不改变外部行为的前提下，优先抽取可测试的纯逻辑段，减少嵌套层级。
- 复杂方法重构必须配套回归测试（单测/必要的 UI 行为验证）。

**备选方案**：通过 `@SuppressWarnings` 临时压制规则。  
**不选原因**：掩盖复杂度问题，后续维护成本继续上升。

### 4) 重复代码治理采用“局部抽取 + 公共化门槛”策略

- 先针对 new code 触发的重复段进行局部抽取，快速降低 `new_duplicated_lines_density`。
- 仅当重复逻辑跨模块且稳定时，再下沉到对应 `core_*` 公共层，避免过度抽象。
- 在 PR 级别对新增重复段设置审查要求，避免重复率反弹。

**备选方案**：一次性全仓去重。  
**不选原因**：超出本变更目标，且容易引入行为回归。

### 5) 覆盖率提升采用“风险点对齐”而非“刷指标”

- 新增测试优先覆盖本次修复涉及的分支与边界条件，尤其是安全/解析/事件处理逻辑。
- 对难以单测的 Android 交互代码，采用可抽取逻辑单测 + 必要 instrumentation 的组合。
- 将覆盖率提升作为修复完成的验收条件之一，而非额外任务。

**备选方案**：仅添加低价值测试提高数字。  
**不选原因**：不能有效防回归，维护成本高。

## Risks / Trade-offs

- [风险] 复杂度重构可能改变事件处理时序（尤其 `NumberPicker`） → [缓解] 保留关键行为路径回归测试，分批提交并逐批验证。
- [风险] 安全加密升级可能影响历史数据兼容 → [缓解] 提供版本兼容读取/迁移逻辑，失败时可观测并安全降级。
- [风险] 去重抽取可能造成不恰当公共依赖 → [缓解] 仅对稳定复用逻辑公共化，遵守模块依赖治理检查。
- [风险] 覆盖率提升引入较多测试维护成本 → [缓解] 聚焦高风险路径，避免为低价值分支编写脆弱测试。
- [风险] Hotspot 审查流程与代码修复节奏不一致 → [缓解] 以“代码合并前完成对应 Hotspot 结论”为门禁。

## Migration Plan

1. 固化基线：以 `.sonarcloud-report` 当前结果作为本次治理输入，建立问题分组与优先级列表。  
2. 第一批：完成安全问题修复（漏洞 + 关键 Hotspot）与 release 配置硬化，并补充对应测试。  
3. 第二批：处理高复杂度与高频规范问题（优先问题密集文件），每批执行本地回归。  
4. 第三批：处理 new code 重复问题并补齐剩余测试，确保质量门阻塞指标达到阈值。  
5. 收敛验证：执行 `./gradlew lint testDebugUnitTest` 等检查并复查 SonarCloud 指标。  
6. 回滚策略：若某批引入行为回归，按批次粒度回滚，保留已验证通过的前序批次。
