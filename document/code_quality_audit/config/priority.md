# 优先级矩阵与打分依据

本文件用于固化统一的优先级评估规则，确保不同负责人对同类问题的 P1/P2/P3 分级尽量一致。

## 1. 固定矩阵（Impact × Effort → P）

> 口径来源：`specs/001-code-quality-audit/spec.md`（FR-011）

| Impact / Effort | 小 | 中 | 大 |
|---|---|---|---|
| 高 | P1 | P1 | P2 |
| 中 | P1 | P2 | P2 |
| 低 | P2 | P3 | P3 |

## 2. Impact 判定参考（高/中/低）

优先考虑“影响面”而非“个人感受”：

- 复用面：是否跨模块/跨功能重复出现（越广影响越高）
- 稳定性/缺陷风险：是否可能引入崩溃、播放失败、数据错乱、ANR 等
- 安全/隐私风险：是否涉及敏感信息、授权、token/cookie、明文存储等
- 维护成本：是否高频修改且难以修改（耦合深、入口多、测试困难）

## 3. Effort 判定参考（小/中/大）

优先考虑“迁移与回归成本”：

- 影响面大小：调用点数量、需要迁移的模块数
- 回归成本：是否需要设备/媒体文件/账号；是否需要长链路验证（投屏/下载/字幕等）
- 不确定性：历史语义不清、外部依赖不稳定、兼容性差异（TV/移动端）
- 风险：重构是否容易引入行为回退、是否需要灰度/开关

## 4. 强制要求（减少主观差异）

- 模块报告中每条 Finding/Task 必须填写：Impact、Effort、以及判定依据（简短即可）
- 若判定为 P1：必须至少给出 1 条可验收的治理任务（Task），并写明验收标准
