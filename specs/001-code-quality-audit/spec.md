# Feature Specification: 代码质量排查与复用治理（分模块）

**Feature Branch**: `[001-code-quality-audit]`  
**Created**: 2026-02-03  
**Status**: Draft  
**Input**: User description: "排查一下当前项目的代码质量，是否存在比如同一个功能存在多个不同版本的实现，对代码冗余、复用性等方面 进行详细排查。因为代码库比较庞大，因此先设计规划排查任务，划分不同的模块进行分别排查。对问题进行 逐 步汇总"

## Clarifications

### Session 2026-02-03

- Q: 每条 Finding 的“证据”最低可追溯颗粒度是什么？ → A: 至少包含文件路径 + 关键符号名（类/方法/函数）
- Q: Finding/Refactor Task 是否需要统一的标识（ID）规则？ → A: 模块内 ID + 全局汇总 ID（模块报告用模块内 ID，全局清单为合并后的条目分配全局 ID）
- Q: 排查维度是否包含安全/隐私？ → A: 包含基础安全/隐私维度（仅聚焦明显高风险项）
- Q: 优先级（P1/P2/P3）如何统一评估？ → A: 影响（Impact）× 工作量（Effort）矩阵定级（高影响低成本优先）
- Q: Impact/Effort 的分档尺度采用哪种？ → A: 3×3：Impact=高/中/低；Effort=小/中/大；并在规格中写死映射到 P1/P2/P3

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 制定可执行的排查规划与分工 (Priority: P1)

作为项目维护者/负责人，我希望先获得一份**可执行的排查规划**：明确排查范围、模块划分、统一的检查维度、产出物模板、以及“逐步汇总”的节奏，从而让团队可以按模块分工并以一致口径开始排查。

**Why this priority**: 代码库较大且跨模块协作成本高，若没有统一规划与口径，排查结果会不可比较、难以汇总，也会导致重复劳动与治理方向摇摆。

**Independent Test**: 仅完成“规划文档”即可独立验收：按文档能完成模块分工，并且每个模块的排查输出格式一致，可直接进入下一阶段排查。

**Acceptance Scenarios**:

1. **Given** 项目已确定纳入排查范围，**When** 打开本规格说明，**Then** 能看到清晰的模块分组、排查维度清单、以及每个模块必须产出的报告结构（可直接用于分工）。
2. **Given** 两位不同维护者分别负责两个模块，**When** 依据同一套维度输出各自的模块报告，**Then** 两份报告的结构一致，能够被统一规则汇总到全局问题清单中。

---

### User Story 2 - 输出模块级排查报告（重复实现/冗余/复用机会） (Priority: P2)

作为模块负责人，我希望能对所负责模块产出一份“模块级排查报告”，识别：
1) 同一功能的多版本/多路径实现（含跨模块重复）  
2) 冗余或低收益的代码（含历史遗留、重复封装、无效分层）  
3) 可复用/可抽取的共性能力与建议的统一落点  
以便后续重构治理时能够快速决策“统一/保留多实现/废弃”。

**Why this priority**: 模块级报告是“逐步汇总”的基本单位；先保证每个模块都有可复用的排查产出，才能建立可靠的全局治理输入。

**Independent Test**: 任选一个模块即可独立验收：该模块报告中包含必需栏目（背景、发现、证据、影响、建议、风险、优先级），并能形成至少 1 条可执行的治理任务。

**Acceptance Scenarios**:

1. **Given** 已确定某模块为排查对象，**When** 完成该模块的排查，**Then** 产出一份模块报告，且报告中每条问题都可追溯到明确的文件路径与关键符号名（并可归属到模块/目录范围）与原因描述（不要求立刻改代码）。
2. **Given** 发现“同一功能存在多个实现路径”，**When** 在报告中记录，**Then** 必须标注其类型：是“有意的多实现策略”还是“无意的重复实现”，并给出保留/统一/废弃的推荐结论与理由。

---

### User Story 3 - 全局逐步汇总与治理路线图 (Priority: P3)

作为技术负责人/项目管理者，我希望把各模块报告逐步汇总为“全局问题清单与治理路线图”，用于决策优先级、拆分治理批次、评估风险与依赖，并持续跟踪收敛。

**Why this priority**: 治理资源有限，需要把“最影响复用/维护成本/缺陷密度”的问题优先处理，同时避免因不一致的模块改动引入依赖环或跨模块耦合。

**Independent Test**: 当至少完成两个模块报告后，即可产出可独立验收的全局汇总：能够列出跨模块重复点、给出优先级排序，并形成一份可执行的治理任务列表。

**Acceptance Scenarios**:

1. **Given** 已存在多个模块报告，**When** 执行汇总，**Then** 形成全局问题清单，并能按“重复实现/冗余/复用机会/架构一致性风险”等维度分类与排序。
2. **Given** 同一类问题在多个模块出现，**When** 形成治理任务，**Then** 任务必须明确“统一落点”（例如归属到某个基础能力层或契约层）与预期边界，且不引入跨特性模块的直接依赖。

---

### Edge Cases

- **有意的多实现**：同一能力存在多种实现是设计选择（例如适配不同运行环境/输入方式/渲染路径）。此类必须记录“保留多实现”的理由与边界，避免被误判为冗余。
- **源码目录存在但未纳入构建**：仓库中可能存在未被当前构建引入的模块/目录；需要明确是否纳入排查或作为“观察项”单独记录。
- **第三方/封装依赖**：外部封装模块（如打包依赖或 AAR 包装）通常不宜做结构性治理，但需要识别其对重复实现与复用边界的影响。
- **Debug/Release 或不同入口差异**：同一功能在不同运行模式下可能有分支实现；需要确认是否属于合理分流，避免误合并导致行为回退。
- **同名但语义不同**：表面相似的工具类/方法名可能承担不同语义；需要要求报告提供“语义差异说明”，避免盲目统一。

## Requirements *(mandatory)*

### Audit Scope & Module Partition

本特性面向“当前构建纳入的模块”进行分模块排查与逐步汇总。建议按分层与协作边界划分批次，先基础能力后业务特性，最后应用壳层：

- **应用壳层（Composition Root）**：`:app`
- **业务特性模块（Feature）**：`:anime_component`、`:local_component`、`:user_component`、`:storage_component`、`:player_component`
- **基础能力模块（Core/Infra）**：`:core_contract_component`、`:core_log_component`、`:core_system_component`、`:core_network_component`、`:core_database_component`、`:core_storage_component`、`:core_ui_component`、`:bilibili_component`
- **数据模型模块（Data）**：`:data_component`
- **封装依赖模块（Repository wrappers）**：`:repository:danmaku`、`:repository:immersion_bar`、`:repository:panel_switch`、`:repository:seven_zip`、`:repository:thunder`、`:repository:video_cache`

说明：
- “封装依赖模块”原则上不做结构性治理（以接口边界与使用方式审视为主），除非其引发明确的重复实现或耦合风险。
- 若存在未纳入构建但实际被引用/拷贝的代码（例如历史目录），应记录为观察项并评估纳入范围。

### Deliverables & Incremental Reporting

- **排查规划**：范围、模块分组、排查维度、报告模板、优先级规则、以及“逐步汇总”的节奏与责任分工。
- **模块级报告（逐模块）**：按统一模板输出该模块的发现与建议，作为全局汇总的输入。
- **全局问题清单（持续更新）**：跨模块去重合并后的问题集合，包含优先级与治理路线图。
- **治理任务列表（Backlog）**：每条高优先级问题对应的可执行任务条目，用于后续重构落地与验收跟踪。

### Functional Requirements

- **FR-001**: System MUST 定义明确的排查范围与模块清单（包含模块分组规则），并说明纳入/不纳入的边界。
- **FR-002**: System MUST 提供统一的“排查维度清单”，至少覆盖：同功能多实现、重复封装/重复工具、低复用/跨模块耦合、历史冗余与可删除项、架构一致性风险，以及基础安全/隐私风险（仅聚焦明显高风险项，如硬编码密钥/Token、敏感信息日志输出、明文存储敏感数据等）。
- **FR-003**: System MUST 定义“模块级排查报告”的标准结构，使不同模块的结果可对齐、可比较、可汇总。
- **FR-004**: System MUST 要求每条发现（Finding）具备可追溯的证据描述（至少包含文件路径与关键符号名，如类/方法/函数；必要时可补充模块/目录/命名空间范围）与影响说明（维护成本/风险/复用价值）。
- **FR-005**: System MUST 对“同一功能存在多个实现”的情况做分类：有意的多实现策略 vs 无意的重复实现，并要求给出治理结论：保留/统一/废弃。
- **FR-006**: System MUST 定义跨模块协作的治理原则：跨特性模块协作不得通过直接依赖完成，应通过契约/路由等既有协作机制；治理方案不得引入依赖环。
- **FR-007**: System MUST 以“逐步汇总”的方式输出全局问题清单：支持随着模块报告的完成持续追加、去重、合并与优先级调整（不要求一次性完成全量排查才可交付价值）。
- **FR-008**: System MUST 为每个治理建议形成可执行的任务条目（Refactor Task），至少包含：目标、范围、预期收益、风险与依赖、验收标准、优先级与负责人。
- **FR-009**: System MUST 明确并记录本次排查的关键假设与依赖（例如：哪些模块属于“基础能力层/特性层/应用壳层”的边界），以保证输出一致性。
- **FR-010**: System MUST 定义 Finding 与 Refactor Task 的标识（ID）规则：模块报告内使用模块内唯一 ID；全局问题清单对跨模块去重合并后的条目分配全局 ID，并保留与来源模块内 ID 的映射关系以便追踪。
- **FR-011**: System MUST 定义统一的优先级评估规则：以影响（Impact）× 工作量（Effort）矩阵进行分级，并明确映射到 P1/P2/P3（高影响低成本优先）。
  - Impact 分档：高 / 中 / 低
  - Effort 分档：小 / 中 / 大
  - P1/P2/P3 映射（写死口径，便于跨模块对齐）：

  | Impact / Effort | 小 | 中 | 大 |
  |---|---|---|---|
  | 高 | P1 | P1 | P2 |
  | 中 | P1 | P2 | P2 |
  | 低 | P2 | P3 | P3 |

### Requirement Acceptance Criteria

- **AC-FR-001**: 存在一份“范围与模块清单”，并显式标记纳入/不纳入项与原因；模块分组规则可被他人复用进行分工。
- **AC-FR-002**: 存在一份“排查维度清单”，且被所有模块报告引用；抽查任意 2 个模块报告，维度口径一致。
- **AC-FR-003**: 存在统一的“模块报告结构”，且每个模块报告包含：背景、发现、证据、影响、建议、风险、优先级/负责人。
- **AC-FR-004**: 抽查任意 10 条 Finding，至少 9 条包含可追溯证据描述（至少包含文件路径与关键符号名）与影响说明（不要求包含具体实现方案）。
- **AC-FR-005**: 任意标记为“多实现”的 Finding 均包含分类（有意/无意）与结论（保留/统一/废弃），且有理由说明。
- **AC-FR-006**: 任意治理建议均不要求特性模块之间建立直接依赖；若需要跨模块协作，必须通过既有协作机制表达（以避免依赖环风险）。
- **AC-FR-007**: 全局问题清单可随模块报告增量更新；新增模块报告不会导致全局汇总结构重写或口径改变。
- **AC-FR-008**: 每条高优先级 Finding 都对应至少 1 条可执行治理任务，且任务包含验收标准与负责人。
- **AC-FR-009**: 规格说明或汇总文档中存在“假设与依赖”清单；当假设变化时，能够定位受影响的模块报告与治理任务。
- **AC-FR-010**: 任意模块报告中的 Finding/Refactor Task 均具备模块内唯一 ID；全局问题清单中的条目具备全局 ID，且可追溯到至少 1 个来源模块内 ID（支持多来源合并）。
- **AC-FR-011**: 抽查任意 10 条 Finding/治理任务，均能给出其 Impact/Effort 评估，并依据统一矩阵得到一致的 P1/P2/P3 分级结果。

### Assumptions & Dependencies

- **Assumption 1**: 排查范围以“当前构建实际纳入的模块”为主；仓库中存在但未纳入构建的目录作为“观察项”单独记录，不阻塞主流程。
- **Assumption 2**: 本次工作以“识别问题 + 形成治理任务”为主要目标；实际代码改动与重构落地作为后续独立阶段推进。
- **Assumption 3**: 对同一能力的多实现，不默认判定为冗余；优先判断其是否为明确的策略/适配需求，并要求在汇总中保留设计意图。
- **Dependency 1**: 需要明确的模块分层与依赖治理规则作为判断依据（避免治理建议引入依赖环或跨特性直连）。
- **Dependency 2**: 需要模块负责人/维护者参与确认“语义差异/历史背景/保留原因”，避免误判导致行为回退。

### Key Entities *(include if feature involves data)*

- **Module Group**: 模块分组（如基础能力层、业务特性层、应用壳层等），用于分工与结果汇总口径统一。
- **Finding**: 排查发现的问题条目，包含模块内唯一 ID（用于模块报告内引用）、影响、证据描述、推荐结论与优先级；在全局汇总中可被分配全局 ID，并保留与来源模块内 ID 的映射关系。
- **Module Report**: 单模块排查报告，包含模块背景、发现列表、治理建议与风险评估。
- **Global Summary**: 全局汇总视图，用于跨模块去重合并、优先级排序与路线图输出；为合并后的条目分配全局 ID 并维护来源映射。
- **Refactor Task**: 可执行的治理任务条目，包含模块内唯一 ID；在全局汇总中可被分配全局 ID，并保留与来源模块内 ID 的映射关系，用于后续落地与验收跟踪。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 排查规划产出后，团队能够在 1 个工作日内完成模块分工（每个模块有明确负责人），且报告模板一致可复用。
- **SC-002**: 覆盖率：纳入范围内的模块 100% 具有“模块报告”或明确的“无需治理/仅观察”结论（不可缺失）。
- **SC-003**: 全局问题清单中，所有高优先级（P1）问题 100% 具备明确治理结论（保留/统一/废弃）与可执行任务条目。
- **SC-004**: 汇总质量：任意抽查 10 条 Finding，至少 9 条具备可追溯证据描述与清晰的影响说明，且结论不会引入跨特性模块的直接依赖关系。
